# Open PR Skill

Creates a feature branch from main and opens a draft pull request on GitHub before
any implementation work begins. This gives the team visibility into planned work and
creates a home for all commits produced by the orchestration pipeline.

## When to Use

- Automatically by the orchestrate skill after the architect's plan is confirmed
- Manually when the user says "open a PR", "create a branch", or "start a PR for this"

## Inputs

The orchestrator (or user) provides:
1. **Feature summary** — a 1-2 sentence description of what is being built/fixed
2. **Plan overview** — the architect's plan (wave structure, task list, key decisions)
3. **Plan type** — one of: `feat`, `fix`, `refactor`, `chore`, `perf`

## Process

### Step 1: Derive Branch Name

Generate a branch name from the feature summary using this format:

```
<type>/<kebab-case-summary>
```

Rules:
- `type` is the plan type (feat, fix, refactor, chore, perf)
- Summary is 3-5 words, kebab-case, derived from the feature description
- Max 50 characters total
- No special characters beyond hyphens
- Examples:
  - `feat/add-streak-tracking`
  - `fix/timer-race-condition`
  - `refactor/extract-audio-manager`
  - `feat/history-chart-weekly-view`

### Step 2: Create Branch

```bash
git checkout -b <branch-name>
```

If the branch already exists (from a prior interrupted session), ask the user whether to
reuse it or create a new one with a numeric suffix (e.g., `feat/add-streak-tracking-2`).

### Step 3: Push Branch and Open Draft PR

Push the empty branch and open a draft PR:

```bash
git push -u origin <branch-name>
```

Then create the PR using `gh pr create` with this structure:

```bash
gh pr create --draft --title "<PR title>" --body "<PR body>"
```

### PR Title Format

```
<type>(<scope>): <imperative summary>
```

- Same conventional-commit format as commit messages
- Max 72 characters
- Scope is the primary module affected
- Examples:
  - `feat(history): add focus streak tracking`
  - `fix(timer): resolve race condition on start`

### PR Body Format

Use this exact template (fill in the sections from the plan):

```markdown
## Summary

<1-3 sentences describing what this PR implements and why>

## Plan

<High-level description of the architect's approach>

### Tasks

<Numbered list of tasks from the plan, grouped by wave>

**Wave 1:**
- [ ] Task 1.1: <name> (<model>)
- [ ] Task 1.2: <name> (<model>)

**Wave 2:**
- [ ] Task 2.1: <name> (<model>)

### Key Decisions

<2-5 bullet points of architectural decisions made by the architect>

## Verification

- [ ] All tasks implemented and committed
- [ ] Code review passed (code-reviewer)
- [ ] Tests passing with >= 90% coverage
- [ ] ORCHESTRATOR.md updated (if architecture changed)

## Coverage

_Will be updated by the orchestration pipeline after tests run._

---

Generated by the orchestration pipeline.
```

### Step 4: Return PR Info

Output the following so the orchestrator can track it:

```
PR_BRANCH: <branch-name>
PR_NUMBER: <number>
PR_URL: <url>
```

## After Implementation

The orchestrate skill handles committing and pushing to the branch. Once all tasks
are complete:
1. The orchestrator pushes remaining commits: `git push`
2. The orchestrator updates the PR body's coverage section with final numbers
3. The orchestrator marks the PR as ready for review: `gh pr ready <number>`

The orchestrator does NOT merge — the user decides when to merge.

## Error Handling

| Scenario | Action |
|----------|--------|
| `gh` CLI not installed | Print: "Install GitHub CLI: brew install gh" and stop |
| Not authenticated | Print: "Run: gh auth login" and stop |
| Branch name conflicts | Ask user: reuse existing or create with suffix |
| Remote push fails | Report error, suggest user check remote config |
| PR creation fails | Report error with gh output, suggest manual creation |
